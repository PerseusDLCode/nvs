** TODO Fix malformed XML output from parse_annotations_to_xml
:PROPERTIES:
:CUSTOM_ID: nvs/parser-malformed-xml-output
:END:
Unblocked — =parsing-raw-xml-vs-plain-text= is decided and
=nvs/parser-refactor-lxml= is merged to =main=. Ready for a fresh
Claude Code session.
Discovered while verifying =annotations/xml-to-annotations= against
real corpus data (=make out/shake.var.lear.annot.xml=):
=parse_annotations_to_xml= (=src/hypervariorum/nvs/parsing.py=) never
escapes inline formatting tags (=<B>=, =<I>=, =<SC>=, =<G>=, =<SUB>=,
=<P>=) from the source transcription — they're written straight into
the generated XML as literal, unescaped markup.

Two distinct failure modes found in =raw/shake.var.lear.txt= alone:
- Numeric/illegible-text codes (=<01>=, =<03>=, =<*>=) — fixed
  separately by converting them to real characters/=<unclear/>= in
  =nvs/entities.py= (this pass).
- =<P>= paragraph breaks that don't match the chunk-splitter's
  lookahead pattern in =parse_annotations_to_xml= (expects a
  digit-lemma or bracketed lemma right after =<P>=) are left
  embedded as literal, unclosed tags inside =<commentary>= — 21
  occurrences in King Lear alone, causing
  =xml.etree.ElementTree.ParseError: mismatched tag=.

Needs design thought before implementation: should inline formatting
tags be escaped as literal text (simplest, loses the formatting
distinction), or modeled as real nested XML content within
=<lemma>=/=<commentary>= (preserves structure, bigger change to both
the writer and to =model/annotation.py='s consumption of it)? Given
the design weight, consider a =doc/forum.org= entry before
implementing, per the collaboration protocol's guidance to link back
to a real design decision.

Settled per
[[file:doc/forum.org::#parsing-raw-xml-vs-plain-text][#parsing-raw-xml-vs-plain-text]]:
- Scope is the eight =shake.var.*= files. =titus.f1.txt= is a
  different source format entirely (First Folio base text, not
  variorum commentary) and is explicitly out of scope — don't include
  it when checking the completion criterion below.
- Build an explicit tag-classification registry (milestone /
  container / inline-formatting-deferred), populated by inventorying
  tags occurring inside =<CC>...</CC>= spans across the eight files
  (not the whole file — =extract_elements= already discards
  everything outside =HE=/=CC=).
- Inline formatting tags (=B=, =I=, =SC=, =G=, =SUB=) are escaped as
  literal text, not modeled as XML structure — deliberately
  provisional; see the Decision for the revisit trigger (tied to
  =bibliography/parser= below).

Completion criterion: =make out/<play>.annot.xml= produces
well-formed XML (=xml.etree.ElementTree.parse= succeeds without
error) for each of the eight =shake.var.*= plays (=titus.f1.txt=
excluded, out of scope), and
=hypervariorum.model.annotation.load_annotations()= successfully
loads each result.

Also in scope, found during retrospective review of
=annotations/xml-to-annotations= (see the Review section above):
=lemma= and =commentary= are written into the output via plain
f-strings with no XML escaping. Any literal =&=, stray =<=, or =>=
in source text — not just the known inline-formatting tags — will
also produce malformed output. Same fix shape as the tag question:
escape literal text content on the way out.
