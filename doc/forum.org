** Ingestion vs. model: where NVS-specific code ends
:PROPERTIES:
:CUSTOM_ID: architecture/ingestion-model-classifier-split
:ASSIGNEES: Cliff
:SOURCE: chat/2026-07-15
:DEPENDS: packaging/src-layout-split
:DISPOSITION: adopt
:END:

Proposed while queuing the =bin/= → =src/= packaging task: does the
project need an architectural seam between code that is specific to
parsing the NVS transcriptions and code that operates on the
annotation model generally (graph construction, the AI-assisted edge
classifiers)? Renaming the repo/package to =hypervariorum= (see
=process/collaboration-protocol-v2= above) already signals that the
project is meant to outgrow any one source corpus; this question asks
whether the code layout should say so too.


**** Research
[2026-07-15] Parsing is inherently tied to the idiosyncrasies of one
source format — =parse_annotations_to_xml= is built entirely around
this transcription's =<HE>=/=<CC>= tag conventions, with regex tuned
to how this editor marked up acts, scenes, and lemma ranges. That
code will always be NVS-shaped. Classification and graph operations
are, in principle, source-agnostic: a classifier deciding that one
annotation glosses a particular passage doesn't need to know whether
the annotation arrived via NVS markup or some other encoding. This is
the same seam as MVP's Corpus → Annotations → Views layering:
source-specific ingestion feeding a format-independent model, with
higher layers unaware of provenance.

Proposed layout:

#+begin_example
src/hypervariorum/
  nvs/                  <- NVS-specific: markup parsing, entity conversion
    entities.py
    parsing.py
  model/  (or graph/)   <- source-agnostic: Annotation, the dependency graph
    annotation.py
  classifiers/          <- source-agnostic: the AI-assisted edge classifiers
#+end_example

=nvs/= stays as parochial as the source material demands; =model/=
and =classifiers/= are where the project's actual research claim to
Greg/Perseus lives — "a source-agnostic annotation-graph-and-
classifier engine, currently fed by an NVS parser" rather than
"some NVS-specific code."

Open question, not resolved here: is the ingestion layer meant to
stay singular (=nvs/= just documents provenance, permanently the only
source), or is a second source corpus genuinely anticipated (another
variorum edition, or non-Shakespeare material) — in which case =nvs/=
is the first of a =sources/= family rather than a fixture. Cheap to
decide now; expensive to restructure into later. Does not need to be
answered by building a second parser today, just by naming the
likelihood.

Actionable for Claude Code (once Decision is recorded):
- Fold this layout into the =packaging/src-layout-split= task in
  =agenda.org= rather than treating it as a second refactor.

**** Decision
<2026-07-15 Wed 13:26> I had been wondering if there should be an
entirely separate package, in its own repo, for the code that
interfaces with raw files, and then another repo for higher-level code
that works with annotation objects.  And I'm answering my own
question: that's getting too complex for this stage of the project.

So at the beginning of the project we will assume that we will be
using the Furness editions of the NVS as source texts. Explicitly,
then: ingestion stays singular for now — =nvs/= names the one source
and documents its provenance, rather than being built out as a
=sources/= family. Revisit only if a second source corpus actually
materializes.

The Furness editions don't have to live in this repository, though;
see =external/raw-corpus-dependency= below for that question.

*** Research
[2026-07-15] Two distinct issues, worth keeping separate:

1. /Portability of the symlink itself/ — if it points to an absolute
   path, it works on this machine and breaks for anyone else
   (including future-Cliff on different hardware). A relative path
   (=../NVS_Shakespeare=) survives as long as the sibling-repo layout
   under =PerseusDLCode/= holds. Worth checking with =readlink raw=
   and fixing if absolute.

2. /Bootstrapping is undocumented/ — nothing in the repo currently
   tells a fresh clone "go fetch =PerseusDLCode/NVS_Shakespeare= as a
   sibling and symlink it to =raw/=." This doesn't need to wait on the
   harder project-config question (output artifact paths, secrets,
   etc. — deliberately deferred, per Cliff, to a separate discussion).
   A paragraph in =README.md= documenting the symlink and its target
   is a minimal, independent fix.

** Parsing raw XML vs treating it as literal text
:PROPERTIES:
:CUSTOM_ID: parsing-raw-xml-vs-plain-text
:ASSIGNEES: Cliff, Claude Chat
:DEPENDS: nvs/parser-malformed-xml-output
:DISPOSITION: adopt
:END:

<2026-07-17 Fri 12:34> Claude Code surfaced a problem in [[file:../src/hypervariorum/nvs/parsing.py][parsing.py]]
(see [[file:../agenda.org::#nvs/parser-malformed-xml-output][the agenda]]). The current implementation treats the input as plain
text and manipulates it with regular expressions; Claude Code is
trying to parse it into XML.  Should we continue to treat the Critical
Commentary sections as plain text or should we try to parse them?

*** Research
I don't think it is possible to parse the input as XML, because it
simply isn't valid XML.  One example: line 2813 in [[file:../raw/shake.var.lear.txt][shake.var.lear.txt]]
begins a comment that extends over a page break. The starting <P> tag
isn't matched with a closing </P> before the <PB>; on line 2837, the
comment continues on the next page with an opening <P>. And there is
no closing </P> for this comment.  In fact, these <P> tags are
actually milestones, not containing elements.

My thought is that we should continue to treat the raw input as text,
not XML.  However, we should attend to the errors Claude Code surfaced
here and verify that our regexes are working properly.

*** Decision
[2026-07-17] Confirmed: treat Critical Commentary as text, not XML,
for the reasons above — =<P>= is a milestone, not a container; this
was never well-formed XML and never will be. Consequences:

1. /Scope/: applies to the eight =shake.var.*= files (the NVS
   variorum editions). =titus.f1.txt= is a different source entirely
   — a First Folio base-text transcription (=<DIV1>=/=<DIV2>=/
   =<HEAD>=/=<STAGE>=/=<LB>=, attributed =<PB>=), not variorum
   commentary, evidently placed in Greg's =raw/= repository for lack
   of a better home. Out of scope for =nvs/parsing.py=; if it's ever
   needed, it wants its own ingestion path, not a stretch of this one.

2. /Milestones/: identify real milestone tags (=P=, =PB=, possibly
   others) explicitly, as a named classification rather than implicit
   regex behavior, so the parser stops expecting close tags for them.
   Inventory tags occurring *inside* =<CC>...</CC>= spans across the
   eight =shake.var.*= files (=extract_elements= already discards
   everything outside =HE=/=CC=, so front-matter tags — =C=, =SHL=
   ("short horizontal line"), and its companions =LHL=/=FHL= ("long/
   full horizontal line") — never reach the parser and aren't
   actionable here) rather than continuing to discover tags one crash
   at a time.

3. /Inline formatting/ (=B=, =I=, =SC=, =G=, =SUB=): escape as literal
   text for now, not modeled as real XML structure. Deliberately
   provisional, not a verdict on the formatting's value — the plan is
   to use exactly this typography (smallcaps, bold, italic) later to
   help identify author names, book titles, and similar bibliographic
   entities within commentary text, which is real interpretive signal,
   not just display styling. Deferring is safe specifically because
   the lxml-based escaping (queued in =nvs/parser-refactor-lxml=) is
   reversible: parse the stored XML, unescape the text, and the
   original transcription codes are recoverable exactly, so nothing is
   lost by waiting.

   Revisit trigger, not left open-ended: when work begins on using
   typographic markers to identify bibliographic entities in
   commentary (connects directly to =bibliography/parser= in
   =agenda.org=), or once real classifier requirements for
   =commentary= content exist — whichever comes first. Until then,
   keep =B=/=I=/=SC=/=G=/=SUB= as an explicit "known, deferred,
   escape" category in the same tag-classification registry used for
   the milestone fix, not lumped in with genuinely unclassified tags.

4. /Addendum, found verifying =nvs/parser-refactor-lxml=/: of the
   eight =shake.var.*= files, =ham2.txt= has zero =<CC>= tags (410
   =<HE>= tags, none of them wrapping commentary). It's the NVS
   Hamlet appendix volume — bibliography, actor interpretations,
   table of contents, essays — marked up with =<C>=, =<T>=, =<R>=,
   =<CE>=, =<SHL>=, etc. Not out of scope the way =titus.f1.txt= is
   (the parser runs on it correctly and produces valid, trivially
   empty output), but structurally a different kind of document from
   the other seven, in the same family as =titus.f1.txt='s mismatch.
   Likely relevant to =bibliography/parser=: if this genuinely is the
   NVS Hamlet bibliography volume, its content — not scattered =CC=
   commentary — may be that task's real source material.
